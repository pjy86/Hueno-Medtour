## 数据库环境策略

### 统一方案：使用 Render PostgreSQL

```
开发阶段（本地调试）：
┌─────────────────┐     ┌─────────────────────────────┐
│  本地 Next.js   │ ──> │  Render PostgreSQL           │
│  (.env.local)   │     │  (开发数据库)                │
└─────────────────┘     └─────────────────────────────┘

正式部署（Render）：
┌─────────────────┐     ┌─────────────────────────────┐
│  Render Web     │ ──> │  Render PostgreSQL           │
│                 │     │  (同一数据库，同一连接)       │
└─────────────────┘     └─────────────────────────────┘
```

### 环境变量配置

**开发环境 `.env.local`：**

```env
DATABASE_URL="postgresql://user:pass@xxx.onrender.com/hntravel_dev?sslmode=require"
NEXT_PUBLIC_SITE_URL="http://localhost:3000"
# 管理后台 JWT 签名密钥（任选其一即可；ADMIN_JWT_SECRET 优先）
NEXTAUTH_SECRET="本地开发请填随机长串"
```

**Render 生产环境（自动注入 + 需在控制台核对）：**

```env
DATABASE_URL="postgresql://user:pass@xxx.onrender.com/hntravel"
NEXT_PUBLIC_SITE_URL="https://your-app.onrender.com"
# 强烈建议在生产 Web Service 的 Environment 中显式添加随机 NEXTAUTH_SECRET（或 ADMIN_JWT_SECRET）
```

**与安全相关的可选变量：**

```env
# 登录接口按 IP 限流（默认值见 README）；仅当需要调紧/放宽时修改
ADMIN_LOGIN_MAX_ATTEMPTS_PER_WINDOW=15
ADMIN_LOGIN_WINDOW_SECONDS=900
```

说明：`NEXTAUTH_SECRET` **不是用来加密数据库里的密码**，而是用来 **签发/校验管理员登录后的 JWT**（配合 HttpOnly Cookie `admin_session`）。若密钥泄露或使用过弱默认值（如字面量 `secret`），攻击者可能在**不需要知道后台密码**的情况下伪造会话，因此生产环境必须使用强随机密钥，或依赖 Render 上的派生兜底（详见下文「后台安全与会话」）。管理员登录密码仍以 **bcrypt** 哈希存于 `admins` 表。

### 工作流程

1. **开发时**：本地连接 Render PostgreSQL，使用 `prisma studio` 查看数据
2. **部署时**：Render 自动使用相同的 `DATABASE_URL`，无需更改代码
3. **数据迁移**：`prisma db push` 同步表结构到数据库

### 可选：双数据库环境

如果需要区分开发和生产数据，可以在 Render 创建两个 PostgreSQL 实例：

- `hntravel_dev` - 开发用
- `hntravel_prod` - 生产用

本地开发时 `.env.local` 指向 `hntravel_dev`，Render 部署时使用 `hntravel_prod`。

```
┌─────────────────────────────────────────────────────────┐
│                      Render 部署                        │
├─────────────────────────────────────────────────────────┤
│  Next.js (前端 + API Routes)                            │
│  └── app/api/cms/*  (CMS API；GET 仅限已登录管理员)       │
│  └── app/api/contact/* (联系表单；POST 公开，GET 仅管理员)│
│  └── app/api/admin/*    (登录、会话、改密；JWT Cookie)       │
├─────────────────────────────────────────────────────────┤
│  PostgreSQL (Render自带)                                │
│  └── contents 表 (首页文案配置)                          │
│  └── images 表 (图片URL配置)                             │
│  └── contacts 表 (客户提交的联系方式)                     │
└─────────────────────────────────────────────────────────┘
```

## 数据库设计

```sql
-- 文案配置表
CREATE TABLE contents (
  id SERIAL PRIMARY KEY,
  key VARCHAR(100) UNIQUE NOT NULL,
  en TEXT,
  zh TEXT,
  id_text TEXT,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 图片配置表
CREATE TABLE images (
  id SERIAL PRIMARY KEY,
  key VARCHAR(100) UNIQUE NOT NULL,
  url TEXT,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 联系方式表
CREATE TABLE contacts (
  id SERIAL PRIMARY KEY,
  name VARCHAR(255),
  email VARCHAR(255),
  phone VARCHAR(100),
  country VARCHAR(100),
  message TEXT,
  age VARCHAR(10),
  gender VARCHAR(50),
  whatsapp VARCHAR(100),
  medicalNeeds TEXT,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

## 部署到 Render 的关键配置

1. **render.yaml** - 定义 Web 服务和 PostgreSQL；其中 `NEXTAUTH_SECRET: generateValue: true` 仅在通过 Blueprint **首次创建**服务时自动生成。若线上服务曾手动改过环境或未继承该变量，须到 **Render Dashboard → 生产 Web Service → Environment** 手动新增随机 `NEXTAUTH_SECRET`。
2. **环境变量** - `DATABASE_URL`、`NEXT_PUBLIC_SITE_URL`、以及建议显式配置的 **`NEXTAUTH_SECRET`**（或 `ADMIN_JWT_SECRET`，优先级更高）。
3. **Pull Request Preview** — 仍为 `NODE_ENV=production`。若控制台未下发密钥，应用会在 `IS_PULL_REQUEST=true`（仅预览）或 `RENDER=true` 且 **`RENDER_SERVICE_ID` 可用**（正式实例亦有）时自动派生 JWT 密钥以便登录后台；详情见 README。生产环境仍 **推荐手写随机 NEXTAUTH**，与预览/兜底逻辑解耦。
4. **构建命令** — `npm install && npm run build`（与 `render.yaml`/服务设置一致）。
5. **启动命令** — `npm start`。
6. **首次部署后** — 按需执行 `npx prisma db push` 与 `npm run db:seed`。

## 后台安全与会话（已实现）

本节对应代码中与管理员相关的安全基线（P0 + P1），便于后续接手与审计。

### 服务端鉴权（P0）

- **写入与敏感读必须经过 JWT**：`PUT /api/cms/update`、`GET /api/contact`（列表）、`GET /api/cms`（供后台拉全量CMS）均需通过 `requireAdmin`。
- **访客前台**不调用上述 GET；多语言页面在服务端通过 [`src/lib/get-cms-data.ts`](src/lib/get-cms-data.ts) 读数据库，避免把 CMS 明文接口暴露在公网。
- **表单提交**：`POST /api/contact` 仍为公开接口。

### 会话与令牌（P1）

- 登录成功后服务端设置 **HttpOnly** Cookie：`admin_session`（`SameSite=Lax`，生产带 `Secure`），前端一律 **`fetch(..., { credentials: 'include' })`**。
- `requireAdmin` 接受 **Cookie JWT** 或 **`Authorization: Bearer`**（脚本/运维工具）。
- **401** 时前端会调用 `logout` 清 Cookie、提示并跳转登录页。
- **`NEXTAUTH_SECRET` / `ADMIN_JWT_SECRET`**：用于签发与校验 JWT，**不是用来防止「密码库泄露」**，而是防止 **会话令牌被伪造**；登录密码仍以 bcrypt 存储。

### 密码与运维

- 新密码规则：**不少于 12 位**（改密接口与 `npm run admin:set-password` 一致）。
- **忘记密码**：无邮件自助重置；运维在能连 `DATABASE_URL` 的机器上执行：  
  `npm run admin:set-password -- <username> <newPassword>`  
  （脚本见 [`scripts/set-admin-password.ts`](scripts/set-admin-password.ts）。）
- **多账号**：`admins` 表支持多条记录，但产品侧未提供「后台里创建管理员」界面；现阶段由运维/数据库写入；**无角色字段**，所有管理员权限相同。

### 登录限流

- 按 **客户端 IP**（`x-forwarded-for` / `x-real-ip`）计数，超限返回 **429**；实现为进程内内存（多实例时每实例单独计数）。
- 可通过 `ADMIN_LOGIN_MAX_ATTEMPTS_PER_WINDOW`、`ADMIN_LOGIN_WINDOW_SECONDS` 调整。

### Render 与环境

- **`NODE_ENV=production` 且无显式密钥** 时：**非 Render** → 503；**Render 且 `RENDER_SERVICE_ID` 可用** → 自动派生长密钥（预览与正式的派生口径不同；正式侧重随部署相对稳定）。仍建议在 Dashboard **显式配置 `NEXTAUTH_SECRET`**。
- **`secret` 字面量**在生产显式配置时会被拒绝，避免已知弱密钥。

## 代码提交流程（Git / Pull Request）

与本仓库配套的详细规则见 [`.cursor/rules/project-workflow.mdc`](./.cursor/rules/project-workflow.mdc)；[`README.md`](./README.md) 中含英文概要。

**默认流程**

- 日常工作在**独立分支**上开发，`push` 后在 GitHub 创建 **Pull Request**，**base（合并目标）为 `main`**，在合并前保持 PR 为 **Open**。
- 生产 Web Service（Render）已开启 **Automatic Pull Request Previews** 时，该 PR 会触发与生产一致的**云端预览构建**，便于在合并前发现问题；本地仍应按要求运行 `npx tsc --noEmit`（见下文「开发标准」）。
- 预览与审查通过后，在 GitHub **将 PR Merge 进 `main`**，由绑定 `main` 的 Render 服务执行**正式上线构建与部署**。

**不推荐**

- 将日常功能改动**直接 push 到 `main`**：会跳过上述 PR / 预览链路，且不利于 Code Review。（直接推 `main` 仍会触发对 `main` 的常规自动部署，但并非「先预览再合并」的流程。）

**例外**

- 仅在任务或协作中**明确要求**，或团队约定的紧急热修等情形下，可直接提交到 `main`**，并注明原因**。

**可选**

- 在 GitHub 为 `main` 配置 **branch protection**，强制经 PR 合并，减少误直达生产。

## 开发标准（适用于所有项目）

### TypeScript 构建时类型检查

**问题背景**：本地开发时 TypeScript 类型错误不会阻止 `npm run dev`，但 `npm run build` 会进行严格类型检查。如果数据库 Schema 更新后 Prisma Client 未重新生成，部署时会产生类型错误。

**解决方案**：在 `package.json` 的 `build` 脚本中，Next.js 构建前先运行 `prisma generate` 和 `tsc --noEmit`。

```json
"build": "prisma generate && tsc --noEmit && next build"
```

**效果**：

- `prisma generate` - 确保 Prisma Client 类型与数据库 Schema 同步
- `tsc --noEmit` - 运行完整的 TypeScript 类型检查，任何错误都会阻止构建
- 构建失败时不会生成无效的 `.next` 输出，避免部署有问题的代码

**适用场景**：

- 所有使用 TypeScript + Prisma 的项目
- 多开发者协作项目（防止未同步的类型定义被提交）
- CI/CD 流程（确保每次部署前类型正确）

**注意事项**：

- `tsc --noEmit` 会检查所有 `.ts` 和 `.tsx` 文件，首次运行可能较慢
- 确保 IDE 的 TypeScript 版本与 `tsconfig.json` 一致
